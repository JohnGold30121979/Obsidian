Патч инфобазы - это зашифрованный файл `.bispatch`, который применяется к уже установленной базе клиента. Он нужен для безопасной доставки изменений структуры БД, метаданных, отчетов, модулей и справочных данных без пересоздания инфобазы.

## 1. Где смотреть патчи

Откройте:

`Конфигуратор -> Патчи инфобазы`

В окне доступны:

- `Собрать патч...` - собрать зашифрованный `.bispatch` из подготовленной папки.
- `Загрузить патч...` - применить `.bispatch` к текущей инфобазе.
- `Обновить` - перечитать историю примененных патчей.

История хранится в таблице `system_patches`. Повторно применить уже установленный патч нельзя.

## 2. Структура папки патча

`manifest.json` можно не создавать вручную. Если его нет, конфигуратор сам сформирует стандартный манифест внутри `.bispatch`:

- `PatchId` будет создан из даты/времени и имени папки.
- `Version` будет рассчитана от текущей версии инфобазы.
- `Name` будет взято из имени папки.
- `Dependencies` будет заполнено базовой зависимостью от текущей версии.

Минимальная папка может содержать только нужные payload-файлы:

```text
schema.sql
metadata.json
```

Дополнительно можно положить:

```text
manifest.json   - необязательный файл; нужен для ручного управления именем, версией и зависимостями
schema.sql      - SQL-изменения структуры БД
metadata.json   - метаданные справочников/документов
reports.json    - отчеты и печатные формы
modules.json    - разделы и привязка объектов к модулям
data.json       - строки таблиц, которые нужно добавить/обновить
```

Пример папки:

```text
patches/
  1.0.1-employees-passport/
    manifest.json
    schema.sql
    metadata.json
```

## 3. Пример manifest.json

```json
{
  "Format": "BIS.Patch",
  "FormatVersion": 1,
  "PatchId": "2026-07-02-employees-passport",
  "Version": "1.0.1",
  "Name": "Паспортные поля сотрудников",
  "Description": "Добавляет паспортные реквизиты в справочник сотрудников.",
  "Author": "BIS",
  "MinAppVersion": "",
  "CreatedAt": "2026-07-02T00:00:00",
  "Dependencies": ["baseline-1.0.0"]
}
```

Правила:

- `PatchId` должен быть уникальным и не меняться после выпуска.
- `Version` становится текущей версией инфобазы после успешного применения.
- `Dependencies` указывают патчи, которые должны быть уже применены.
- Для первой клиентской базы обычно есть базовая запись `baseline-1.0.0`.

## 4. Пример schema.sql

SQL выполняется внутри транзакции применения патча.

```sql
ALTER TABLE "catalog_employees"
ADD COLUMN IF NOT EXISTS "passport_number" varchar(80);

ALTER TABLE "catalog_employees"
ADD COLUMN IF NOT EXISTS "passport_issued_by" varchar(300);

ALTER TABLE "catalog_employees"
ADD COLUMN IF NOT EXISTS "passport_issue_date" timestamp;
```

Для обычного добавления полей через `metadata.json` система теперь сама добавляет недостающие колонки. `schema.sql` нужен для сложных операций: переименование колонок, перенос данных, индексы, исправление старых значений.

## 5. Форматы payload-файлов

`metadata.json`:

- JSON-массив объектов `MetadataObject`.
- При совпадении по `ObjectType + Name` существующие метаданные обновляются.
- Новые колонки таблицы добавляются автоматически.

`reports.json`:

- JSON-массив объектов `Report`.
- Отчет обновляется по `Code`, если он заполнен, иначе по `Name`.

`modules.json`:

```json
{
  "Modules": [],
  "ModuleItems": []
}
```

`data.json`:

```json
[
  {
    "TableName": "catalog_positions",
    "Mode": "Upsert",
    "Rows": [
      {
        "Id": "00000000-0000-0000-0000-000000000001",
        "code": "DIR",
        "name": "Директор",
        "is_active": true
      }
    ]
  }
]
```

Режимы:

- `Upsert` - добавить строку или обновить по `Id`.
- `Replace` - сначала очистить таблицу, потом вставить строки.

## 6. Как собрать патч у разработчика

1. Подготовьте папку патча с `manifest.json` и нужными файлами. Если `manifest.json` не нужен, можно положить только `schema.sql`, `metadata.json`, `reports.json`, `modules.json` или `data.json`.
2. Откройте `Конфигуратор -> Патчи инфобазы`.
3. Нажмите `Собрать патч...`.
4. Выберите папку патча.
5. Проверьте параметры патча в окне `Параметры сборки патча`.
6. При необходимости измените `Version`, `PatchId`, наименование, описание и зависимости.
7. Выберите путь сохранения файла `.bispatch`.
8. После успешной сборки передайте клиенту только `.bispatch`.

Файл `.bispatch` зашифрован. Вручную менять его нельзя.

## 6.1. Как выпустить версию 2.0.0

Сценарий нужен, когда выпускается крупная сборка, а не обычный маленький патч.

1. Подготовьте папку патча, например `2.0-main-release`.
2. Положите туда нужные файлы: `schema.sql`, `metadata.json`, `reports.json`, `modules.json`, `data.json`.
3. Откройте `Конфигуратор -> Патчи инфобазы -> Собрать патч...`.
4. Выберите папку `2.0-main-release`.
5. В окне `Параметры сборки патча` укажите: `Version = 2.0.0`
6. При желании измените: `PatchId = 2026-07-02-release-2-0-0`
7. Проверьте `Dependencies`. По умолчанию система подставляет последний примененный `PatchId`. Обычно это правильно: версия 2.0.0 должна ставиться поверх последней версии клиента.
8. Нажмите `Продолжить`.
9. Сохраните файл, например `2026-07-02-release-2-0-0.bispatch`.
10. Примените этот файл на тестовой копии старой базы.
11. После проверки передайте `.bispatch` клиенту.

Если нужно разрешить установку 2.0.0 на любую базу без проверки предыдущего патча, очистите поле `Зависимости`. Делать так стоит только осознанно.

## 7. Как применить патч у клиента

1. Перед применением сделайте backup базы PostgreSQL или выгрузку инфобазы.
2. Откройте клиентскую инфобазу.
3. Зайдите в `Конфигуратор -> Патчи инфобазы`.
4. Нажмите `Загрузить патч...`.
5. Выберите файл `.bispatch`.
6. Проверьте информацию о патче и зависимости.
7. Подтвердите применение.
8. После успеха проверьте, что в истории статус `Applied`.
9. Перезапустите рабочее окно, если менялись меню, модули или метаданные.

## 8. Что проверять после применения

- В окне выбора инфобаз отображается новая версия патча.
- В `Патчи инфобазы` есть запись со статусом `Applied`.
- Новые поля появились в метаданных и в таблицах БД.
- Открываются измененные справочники, документы и отчеты.
- Если патч меняет проводки или отчеты, проверить один тестовый документ и один отчет.

## 9. Правило выпуска

Перед отправкой клиенту:

1. Применить патч на локальной копии старой базы.
2. Проверить запуск рабочего режима и конфигуратора.
3. Проверить сценарии, которые патч меняет.
4. Только после этого передавать `.bispatch` клиенту.
