# Журнал проводок `doc_postings` — подробное описание

## Назначение

`doc_postings` — это таблица в PostgreSQL, которая хранит бухгалтерские проводки, сформированные из двух источников:
1. **Основной источник**: таблица `Postings` (EF Core), которая заполняется при проведении документов через `MetadataPostingRule`
2. **Legacy-источник**: `doc_postings` — либо внешняя таблица изFoxPro, либо таблица для хранения импортированных проводок

## Модель данных

Таблица `doc_postings` содержит поля:
- `posting_date` — дата проводки
- `doc_number` — номер документа
- `document_type` — тип документа
- `module_code` — код модуля
- `debit_account` — счет дебета
- `credit_account` — счет кредита
- `amount_kgs` — сумма в основной валюте
- `amount_currency` — сумма в валюте
- `currency_id` — валюта
- `description` — содержание/примечание
- `organization_id` — организация (FK)
- `employee_id` — сотрудник (FK)
- `is_active` — признак активности

## Как работает `PostingService`

### Метод `GetAllPostingsAsync`

**Шаг 1: Загрузка справочных данных**
- Организации → `organizationMap` (Guid → название)
- Сотрудники → `employeeMap` (Guid → название)
- План счетов → `accountMap` (код → "код - название")
- Модули → `moduleLookup` (код/название → модуль)

**Шаг 2: Чтение из `doc_postings`**
```sql
SELECT posting_date, doc_number, document_type, module_code,
       debit_account, credit_account, amount_kgs, amount_currency,
       currency_id, description, organization_id, employee_id
FROM doc_postings
WHERE is_active = true
  AND DATE(posting_date) >= :startDate
  AND DATE(posting_date) <= :endDate
```

Каждая строка преобразуется в `PostingViewModel` с разрешением ссылок:
- `ResolveAccount` → расшифровка счета через план счетов
- `ResolveReference` → расшифровка организации/сотрудника
- `ResolveModuleName` → определение модуля по коду или типу документа

**Шаг 3: Чтение из `DynamicDocuments`**
- Загружаются все динамические документы
- Для каждой строки (`DynamicDocumentRow`) извлекается JSON `Data`
- Из JSON извлекаются поля: `SUMMA`, `DEBET`, `KREDIT`, `TEX`, `ORGANIZATION` и русские аналоги
- Если дебет или кредит заполнен — создается проводка

**Шаг 4: Объединение и сортировка**
- Все проводки объединяются в один список
- Сортировка по дате (↓) и номеру документа

## Использование в UI

### `DocumentPostingsDialog`
- **Назначение**: просмотр всех проводок конкретного документа
- **Вызов**: из `AccountingReportsView` (двойной клик по счету) или из документов
- **Отображение**: таблица с колонками Дата, Дебет, Расшифровка дебета, Кредит, Расшифровка кредита, Сумма, Содержание
- **Итого**: количество проводок и общая сумма

### `TurnoversDialog`
- **Назначение**: детализация оборотов по конкретному счету за период
- **Данные**: `PostingService.GetTurnoversByAccountAsync`
- **Отображение**: таблица с дебетом/кредитом по документу

## Интеграция с модулями

### 1. Финансы (кассовые ордера, платежные поручения)
- При проведении кассового ордера создаются проводки:
  - ПКО: Дт счета кассы / Кт корреспондирующего счета
  - РКО: Дт корреспондирующего счета / Кт счета кассы

### 2. Учет материальных ценностей
- Приход/расход ТМЦ → проводки по счетам 10, 41, 43 и т.д.

### 3. Основные средства
- Ввод ОС, амортизация → проводки по счетам 01, 02, 20 и т.д.

### 4. Импорт из FoxPro
- Документы, импортированные из DBF, хранятся в `DynamicDocument` + `DynamicDocumentRow`
- Проводки извлекаются из JSON-полей строк

## Особенности

- **Два источника**: универсальный подход к данным из разных систем
- **Нормализация номеров**: `MetadataService.NormalizeLegacyDocumentNumber` для совместимости
- **Разрешение ссылок**: динамическая подстановка названий по ID
- **Фильтрация по дате**: используется во всех методах получения проводок
- **Активность**: `is_active = true` — только активные проводки
