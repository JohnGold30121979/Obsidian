# Правила распознавания FoxPro-полей в FRX

  

Этот документ описывает, как настраивать общие правила распознавания FoxPro-полей для импортированных FRX-отчетов в визуальном построителе BIS ERP.

  

Главная идея: один раз связать старое FoxPro-поле или выражение с нормальным полем текущей системы, сохранить это как правило, и потом применять автоматически к похожим отчетам.

  

## Где настраивать

  

1. Откройте `Конфигуратор`.

2. Перейдите в раздел отчетов и откройте нужную печатную форму FRX в редакторе.

3. Откройте вкладку `FRX поля`.

4. В верхней таблице `Соответствие элементов макета полям данных` выберите текущие поля данных для элементов FRX.

5. Ниже находится блок `Общие правила распознавания FoxPro-полей`.

  

На вкладке есть три основные кнопки:

  

| Кнопка | Что делает |

| --- | --- |

| `Применить общие правила` | Берет уже сохраненные правила и пытается автоматически заполнить пустые привязки элементов FRX. |

| `Сохранить привязки как правила` | Берет текущие вручную выбранные привязки и превращает их в общие правила для будущих похожих отчетов. |

| `Сохранить список правил` | Сохраняет изменения в нижней таблице правил: ручные добавления, удаления, правки приоритета, regex и активности. |

  

## Что такое правило

  

Правило отвечает на вопрос:

  

> Если в FRX встречается старое поле FoxPro `X`, какое поле текущей системы нужно подставить вместо него?

  

Правило состоит из таких колонок:

  

| Колонка | Значение |

| --- | --- |

| `FoxPro поле/выражение` | Старое имя поля или выражение из FRX. Например `name_kod`, `VED.NAME_KOD`, `debsum`, `kod_arm`, `DTOC(DATE)`. |

| `Каноническое поле` | Внутренний смысл поля. Например `operation_name`, `debit_amount`, `module`. |

| `Поле данных` | Реальное поле текущего отчета BIS ERP, которое будет подставляться. Например `operation_name` или `Модуль`. |

| `Профиль` | Код профиля распознавания. Для актов сверки обычно `foxpro_reconciliation_act`. |

| `Приоритет` | Чем меньше число, тем раньше правило применяется. Обычно достаточно `100`. |

| `Regex` | Если включено, `FoxPro поле/выражение` трактуется как регулярное выражение. |

| `Активно` | Если выключить, правило хранится, но не применяется. |

| `Комментарий` | Заметка для себя и команды. |

  

## База знаний распознавания

  

Общая база знаний находится в классе:

  

[FoxProReportKnowledgeBase.cs](E:/C#/BIS.ERP/BIS.ERP/Services/FoxProReportKnowledgeBase.cs)

  

Она содержит встроенные правила нормализации и алиасы. Это нужно, чтобы не создавать отдельное правило на каждую вариацию одного и того же поля.

  

### Что база знаний делает автоматически

  

Она нормализует выражения FoxPro:

  

| FRX выражение | Нормализуется к |

| --- | --- |

| `{name_kod}` | `name_kod` |

| `=name_kod` | `name_kod` |

| `ALLTRIM(name_kod)` | `name_kod` |

| `DTOC(date)` | `date` |

| `VED.NAME_KOD` | `VED.NAME_KOD`, затем понимает базовое поле `NAME_KOD` |

| `DB_CR.DATOBR` | `DB_CR.DATOBR`, затем понимает базовое поле `DATOBR` |

  

Она понимает старые префиксы таблиц FoxPro:

  

| Префикс | Откуда обычно |

| --- | --- |

| `ved`, `ved2`, `ved3`, `ved4` | Ведомости, обороты, промежуточные таблицы |

| `db_cr`, `db_crs`, `dbcr`, `dbcrs` | Данные проводок и остатков |

| `ksprorg` | Справочник организаций |

| `avt_p` | Справочник авансовых платежей / пар счетов |

| `fact`, `curFACTSW`, `irfactsw` | Счет-фактуры |

  

Она добавляет алиасы для строк отчета:

  

| Пример алиаса | Для чего нужен |

| --- | --- |

| `line1_name_kod` | Первая строка отчета |

| `line1.name_kod` | Первая строка отчета в стиле FoxPro |

| `ved.name_kod` | Старое поле из VED |

| `ved3_debsum` | Старое поле из VED3 |

| `db_crs.korsch` | Старое поле из DB_CRS |

  

## Канонические поля

  

Каноническое поле - это нормальный внутренний смысл старого FoxPro-поля.

  

### Акт сверки

  

| Каноническое поле | Старые FoxPro-алиасы | Текущие поля BIS ERP |

| --- | --- | --- |

| `operation_name` | `name_kod`, `naim`, `name`, `operation`, `material_name` | `Наименование материала, вид операции`, `operation_name` |

| `debit_account` | `schet`, `deb`, `debet`, `debit` | `Дебет`, `debit_account` |

| `credit_account` | `kor_sch`, `korsch`, `cred`, `credit` | `Кредит`, `credit_account` |

| `debit_amount` | `debsum`, `sum_deb`, `deb_beg`, `deb_v` | `Сумма Дт`, `debit_amount` |

| `credit_amount` | `credsum`, `sum_cred`, `cred_beg`, `cred_v` | `Сумма Кт`, `credit_amount` |

| `document_number` | `dok`, `dokum`, `nom_dok`, `doc` | `N докум`, `document_number` |

| `document_date` | `date`, `datobr`, `doc_date` | `Дата`, `document_date` |

| `module` | `module`, `prs`, `kod_arm` | `Модуль`, `module` |

  

Важно: старое `АРМ`, `kod_arm`, `prs` в интерфейсе и новых правилах считаем `Модуль`.

  

## Самый простой способ создать правила

  

Этот способ лучше использовать почти всегда.

  

1. Импортируйте или откройте FRX-форму.

2. Откройте вкладку `FRX поля`.

3. В верхней таблице найдите элемент FRX, например `name_kod`.

4. В колонке `Поле данных` выберите `Акт сверки - наименование операции`.

5. Для `debsum` выберите `Акт сверки - сумма Дт`.

6. Для `credsum` выберите `Акт сверки - сумма Кт`.

7. Для `kod_arm` или `prs` выберите `Модуль`.

8. Нажмите `Сохранить привязки как правила`.

9. Нажмите `Сохранить список правил`, если дополнительно правили нижнюю таблицу.

  

После этого похожий FRX можно открыть и нажать `Применить общие правила`.

## Пример regex-правила

  

Regex нужен, если в разных FRX одно и то же поле записано с разными префиксами.

  

Например:

  

| Колонка | Значение |

| --- | --- |

| `FoxPro поле/выражение` | `(?i)^(?:ved\d*|db_crs?|dbcrs?)[._]name_kod$` |

| `Каноническое поле` | `operation_name` |

| `Поле данных` | `operation_name` |

| `Профиль` | `foxpro_reconciliation_act` |

| `Приоритет` | `50` |

| `Regex` | включено |

| `Активно` | включено |

  

Такое правило поймает:

  

| Выражение |

| --- |

| `ved.name_kod` |

| `ved2.name_kod` |

| `ved3_name_kod` |

| `db_cr.name_kod` |

| `db_crs_name_kod` |

  

Если можно обойтись обычным правилом без regex, лучше использовать обычное правило. Оно проще и безопаснее.

  

## Как применять правила к новому FRX

  

1. Откройте новый FRX-отчет в конфигураторе.

2. Перейдите на вкладку `FRX поля`.

3. Нажмите `Применить общие правила`.

4. Проверьте верхнюю таблицу соответствий.

5. Если часть полей не распозналась, заполните их вручную.

6. Нажмите `Сохранить привязки как правила`.

7. Сформируйте отчет и откройте предпросмотр.

  

## Как понять, что правило сработало

  

В верхней таблице `Соответствие элементов макета полям данных` у элементов FRX должна заполниться колонка `Поле данных`.

  

Например:

  

| Элемент FRX | Поле данных после применения |

| --- | --- |

| `name_kod` | `Акт сверки - наименование операции` |

| `schet` | `Акт сверки - дебет` |

| `korsch` | `Акт сверки - кредит` |

| `debsum` | `Акт сверки - сумма Дт` |

| `credsum` | `Акт сверки - сумма Кт` |

| `kod_arm` | `Модуль` |

  

Если поле не заполнилось, проверьте:

  

| Что проверить | Что должно быть |

| --- | --- |

| Правило активно | `Активно` включено |

| Имя старого поля | В `FoxPro поле/выражение` нет лишних пробелов и опечаток |

| Поле данных | Выбрано реальное поле текущего отчета |

| Regex | Включен только если шаблон действительно regex |

| Приоритет | Более точное правило имеет меньший приоритет |

## Как правила используются при предпросмотре

  

Когда открывается предпросмотр FRX:

  

1. Система берет текущий расчетный отчет BIS ERP.

2. Загружает активные правила из таблицы `FoxProReportFieldRules`.

3. Создает набор алиасов для старых FoxPro-имен.

4. При рендеринге выражений FRX ищет значение по этим алиасам.

5. Если правило говорит, что `kod_arm` равно `module`, в макет попадет текущий модуль.

  

Поэтому правило влияет не только на визуальный построитель, но и на печатный предпросмотр.

  

## Когда добавлять в код, а когда через конфигуратор

  

Через конфигуратор добавляем:

  

| Ситуация | Пример |

| --- | --- |

| В одном FRX поле называется необычно | `SUMMA_D`, `N_DOK`, `ORG_NAME` |

| Нужно быстро сопоставить поле без изменения программы | `VED3.MY_FIELD -> operation_name` |

| Правило относится к конкретной группе отчетов | Только акты сверки |

  

В код `FoxProReportKnowledgeBase` добавляем:

  

| Ситуация | Пример |

| --- | --- |

| Алиас системный и встречается во многих отчетах | `kod_arm/prs -> module` |

| Нужно научить систему новой обертке FoxPro | Новая функция вроде `NVL(...)` |

| Нужно поддержать новый тип старой таблицы | Новый префикс набора данных |

  

То есть конфигуратор - для рабочих правил и быстрых правок, кодовая база знаний - для фундаментальных правил миграции FoxPro.

  

## Рекомендации

  

| Рекомендация | Почему |

| --- | --- |

| Сначала делайте привязку вручную в верхней таблице, потом нажимайте `Сохранить привязки как правила`. | Так меньше риска ошибиться в имени текущего поля. |

| Не включайте `Regex`, если правило можно записать обычным именем поля. | Обычные правила проще отлаживать. |

| Используйте `module`, а не `arm`. | В новой системе это модуль, не АРМ. |

| Для сумм используйте `debit_amount` и `credit_amount`. | Это понятнее, чем старые `debsum/credsum`. |

| Для даты документа используйте `document_date`. | Тогда `date`, `datobr`, `DTOC(date)` можно свести к одному смыслу. |

| Для спорных правил ставьте больший приоритет, например `200`. | Более точные правила с приоритетом `50` или `100` применятся раньше. |
